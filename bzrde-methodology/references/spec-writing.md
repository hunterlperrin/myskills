# Spec Writing — 計画・設計・仕様を書く

Planningフェーズで読む。にからすの最重要手法。3層のアウトプットに対応する:
- **層1: プロジェクト計画**（ロードマップ、WBS、フェーズ設計）
- **層2: 事業計画/戦略**（市場分析、GTM、収支モデル、Go/No-Go判定）
- **層3: システム/ソフトウェア仕様**（API設計、DB設計、UI仕様）

どの層でも、プロセスは同じ6ステップ。テンプレートが異なる。

## インターフェース

**入力**: タスク定義書（確定版）、リサーチ結果、確定情報リスト
**出力**: 仕様書（6点セットから案件に応じて選択）、論点リスト（残🅿️付き）
**次のフェーズ**: Execution（実装開始）or Research（確定率不足時）

---

## 1. 6ステップ解析プロセス（全層共通）

### Step 1: 原文保持で全情報抽出
インプットから解釈を加えずに全情報を抜き出す。

### Step 2: 事前論点リストと照合
成果物（仕様書）から逆算して確認すべき項目を網羅的に作成し、抽出結果と照合。

### Step 3: 確信度タグ付け
確定 / 仮説 / 不明 / 🅿️ を全抽出情報に付ける。

### Step 4: 不足情報の特定
「不明」「🅿️」の項目を一覧化。誰に何を確認するか特定。

### Step 5: 構造化
確定情報を仕様書テンプレートに流し込む。仮説は仮説マーク付きで流し込む。

### Step 6: 品質検証
曖昧語がないか、確定率は80%以上か、AIが一発で実行できる具体性があるか。

---

## 2. 仕様書6点セット

案件の規模に応じて選択する。全て作る必要はない。

### product.md（プロダクト概要）
```markdown
# [プロダクト名]
## ビジョン（1-2文）
## ターゲットユーザー
## 主要機能
| # | 機能 | 概要 | 優先度 |
## 成功指標
| KPI | 現状 | 目標 | 計測方法 |
## 制約条件
```

### requirements.md（要件定義）
```markdown
## 機能要件
| ID | 要件 | 優先度 | 確定状態 |
| FR-01 | [EARS形式] | Must/Should/Could | 確定/仮説/🅿️ |

## 非機能要件
| ID | カテゴリ | 要件 | 基準値 |

## 確定率: [XX/YY項目確定 = ZZ%]
```

### design.md（設計仕様）
```markdown
## アーキテクチャ概要
## コンポーネント設計
### [コンポーネント名]
- 責務 / 入力 / 出力 / 事前条件 / 事後条件 / 不変条件
## データモデル
## API設計
| エンドポイント | メソッド | 入力 | 出力 | 説明 |
```

### tasks.md / structure.md / kpi-definitions.md
タスク分解、ファイル構成、KPI定義。テンプレートは案件に応じて使う。

## 3. EARS形式の5パターン

要件記述のゴールドスタンダード。曖昧さゼロを実現する。

| パターン | 形式 | 例 |
|----------|------|-----| 
| Ubiquitous | The system shall [action] | The system shall encrypt all stored passwords using bcrypt |
| Event-driven | When [event], the system shall [action] | When user submits login form, the system shall validate within 2s |
| State-driven | While [state], the system shall [action] | While in maintenance mode, the system shall display notice |
| Optional | Where [condition], the system shall [action] | Where user has admin role, the system shall display admin panel |
| Unwanted | If [unwanted], the system shall [action] | If DB connection fails, the system shall retry 3 times |

全機能要件をこの5パターンのいずれかで書く（層3のみ必須、層1-2は推奨）。

## 4. 論点リスト

```markdown
| # | 論点 | 仮説 | 確認方法 | 確認者 | 結果 | 確定状態 |
|---|------|------|---------|--------|------|---------|
| 1 | [質問形式] | [推測] | [誰にどう聞くか] | [名前] | [回答] | 確定/🅿️ |
```

確定率を常に追跡: 「10/15項目確定 = 67%」。80%未満で仕様書を「完成」と呼ぶな。

### 盲点探索

論点リスト作成後に、意識的に以下を問う:
- 「このプロジェクトで議論されていないが、成功に必要な論点はないか？」
- 「クライアントが当日聞いてきそうだが、こちらが準備していない質問はないか？」
- 「技術的・契約的・運用的に、見落としている前提条件はないか？」

盲点で発見した項目は論点リストに🅿️として追加し、確認方法を特定する。

## 5. 層1: プロジェクト計画テンプレート

```markdown
# [プロジェクト名] 計画書
## 1. プロジェクト概要（究極目的 / 直接目的 / スコープ）
## 2. 成果物（成果物名 / 形式 / 完成基準 / 担当 / 期限）
## 3. フェーズ設計（Phase / 期間 / 目標 / 完了条件）
## 4. リスクと対策（リスク / 影響度 / 発生確率 / 対策）
## 5. ステークホルダー（名前 / 役割 / 関与レベル）
## 6. 意思決定フロー
```

---

## 6. 層2: 事業計画/戦略テンプレート

```markdown
# [事業/戦略名] 計画書
## 1. エグゼクティブサマリ（3行で全体像）
## 2. 市場分析（ターゲット市場 / 競合環境 / 参入機会）
## 3. 提供価値（何を、誰に、いくらで、どう）
## 4. 収支モデル（項目 / 月額 / 年額 / 前提条件）
## 5. GTM戦略（初期顧客獲得→スケール→安定化）
## 6. ロードマップ（期間 / マイルストーン / 成功指標）
## 7. Go/No-Go判定基準
## 8. リスクと対策
```

## 具体例

**案件**: チビックシステム IPO 2028ロードマップ
**層**: 層1（プロジェクト計画）+ 層3一部（システムアーキテクチャ）
**6点セットの選択**: product.md + requirements.md + tasks.md（design.mdは次フェーズ）
**6ステップ実行結果**: 確定率 23/28 = 82% → 残🅿️5項目はCTO確認待ち
**出力**: 3フェーズ計画（Phase 1: 基盤整備、Phase 2: 機能拡張、Phase 3: IPO準備）

## よくある失敗

| パターン | 帰結 | 修正策 |
|---------|------|--------|
| 曖昧語を要件に含める | 実装者の解釈がバラつく | 「適切に」→具体的な数値/条件に置換 |
| 確定率80%未満で「完成」と呼ぶ | 実装段階で手戻り多発 | 80%は閾値。未達ならResearchに戻る |
| テンプレートに合わない案件をテンプレートに押し込む | 不自然な構成、重要項目の漏れ | テンプレートはガイド。案件に合わせて変形する |
| 全6点セットを必ず作ろうとする | 小規模案件に過剰な工数 | 案件規模で選択。小規模ならproduct.md + tasks.mdで十分 |

## このフェーズでやらないこと

- 実装、コーディング、デザイン作業（Executionフェーズ）
- 追加のリサーチ（確定率不足ならResearchに戻る）
- テンプレートへの盲従（案件が求める形を優先する）

## 品質基準

- 受け手が追加の質問なしで次のアクションに着手できるレベルの具体性
- 曖昧語ゼロ: 「適切に」「正しく」「必要に応じて」が一切ない
- 確定率80%以上。残りは🅿️として明示
- 層3のみ追加: 全機能要件がEARS 5パターンで記述されている
