# Core Principles — 全タスク共通原則

全てのタスクで最初に読む。にからすの仕事の判断基準の全て。

---

## 1. タスク定義書

全てのタスクはタスク定義書から始まる。定義書なしに動くな。

### テンプレート

```markdown
# タスク定義書: [タスク名]

## 1. 目的
### 究極目的（なぜやるのか。プロジェクト全体での位置づけ）
[記述]

### 直接目的（今回のタスクで何を達成するか）
[記述]

## 2. 成果物
| # | 成果物 | 形式 | 説明 |
|---|--------|------|------|
| 1 | [名前] | [md/docx/pptx等] | [何が書かれているか] |

## 3. 成果物が満たすべき要件
- R-1: [具体的な要件。曖昧語禁止]
- R-2: [同上]

## 4. 要件を満たしているかの確認方法
- R-1: [チェック方法]
- R-2: [チェック方法]

## 5. インタラクションポイント（IP）設計
| IP# | タイミング | エージェントの出力 | ユーザーの役割 |
|-----|-----------|------------------|----------------|
| IP-0 | タスク定義完了時 | タスク定義書 | 計画の承認 |

## 6. タスクステップ
| Step | 内容 | 入力 | 出力 |
|------|------|------|------|
```

### 原則

- 究極目的と直接目的を必ず区別する
- 成果物は「何を」「どんなフォーマットで」「何がどう書かれていればOKか」まで定義
- 要件から曖昧語を排除する。「〜的な」「〜など」「適切に」は禁止
- タスク定義書自体もIP-0としてユーザーの承認を得る

### 定型的な確認質問

タスクが依頼されたとき、以下を自問する:
1. 究極目的は何か？
2. 直接目的は何か？
3. 最終成果物は何か？
4. 成果物の要件は何か？
5. 確認方法は何か？
6. IPをどこに置くか？

## 2. 仮説ベース情報処理

### 仮説を先に立てる

調査や分析の前に、答えを予測する。間違っていてもいい。仮説があると何を探すべきか明確になり、結果を「支持/修正/棄却」で判定できる。

### 原文保持優先

解釈は後回し。まず原文をそのまま抽出する。書き起こし、ドキュメント、インタビュー — 全てにおいて、まず原文を保持してから解釈に入る。

### 確信度タグ

全ての情報に確信度タグを付ける:
- **確定**: ソースで明示的に確認済み
- **仮説**: 推論に基づく。検証が必要
- **不明**: 情報不足。確認が必要
- **🅿️**: 後日確認が必要

### 確定率の追跡

「XX/YY項目確定」で進捗を可視化する。確定率が低いまま次のPhaseに進むな。

## 3. 品質管理

### 品質宣言→検証サイクル

```
1. 宣言: 「このPhaseの品質基準: [1-3文で記述]」
2. 実行: Phase内の作業を行う
3. 検証: 「宣言に対する自己検証: [各基準のPASS/FAIL]」
4. 修正: FAILがあれば修正してから次Phaseへ
```

宣言なしで着手するな。検証なしで次に進むな。品質基準に達していないものを「まあいいか」で通すな。

## 4. IP設計

| IP# | タイミング | エージェントの出力 | ユーザーの役割 |
|-----|-----------|------------------|----------------|
| IP-0 | タスク定義完了時 | タスク定義書 | 計画の承認 |
| IP-n | 中間成果物完成時 | ドラフト | レビュー・修正指示 |
| IP-final | 最終成果物完成時 | 完成版 | 最終承認 |

フェーズの切れ目で必ずユーザー確認を入れる。確認なしで次のフェーズに進まない。確認は「はい/いいえ」で答えられる形で提示する。

---

## 5. 成果物逆算

```
最終成果物のフォーマット確定
  → 各セクションに必要な情報を特定
  → その情報を得るための論点を導出
  → 論点に答えるための調査/ヒアリングを設計
  → 情報収集 → 成果物に反映
```

成果物の形が決まらないままリサーチを始めると、情報は集まるが使えない。

---

## 6. AIへの指示原則

- **成果物から逆算して指示を設計する**: 「〇〇を調べて」ではなく「〇〇を判断するために、△△を□□の形式で調べて」
- **盲点探索を明示的に求める**: 「論点リスト外で重要な情報がないか探せ」
- **品質基準に達するまでやらせる**: 「R-1〜R-3を全て満たすまでイテレーションしろ」
- **日本語の書き起こし入力に対応する**: にからすのインプットは音声書き起こしが多い。句読点なし、重複あり、言い直しありの入力を正しく解釈する

## 7. SDD思想との接続

にからすの方法論はSpec-Driven Development（SDD）と同じ思想を持つ。

| SDD概念 | にからす対応 |
|---------|------------|
| Spec = Source of Truth | タスク定義書 = 全ての起点 |
| .sdlc/context/ | .context/（Project Zero規約） |
| .sdlc/specs/ | 仕様書6点セット |
| .sdlc/templates/ | PJ固有上書き |
| requirements → plan → tasks | Discovery → Planning → Execution |

.context/ディレクトリにプロジェクトの永続的コンテキスト（概要、アーキテクチャ、規約）を配置し、仕様書をソースオブトゥルースとして扱う。
