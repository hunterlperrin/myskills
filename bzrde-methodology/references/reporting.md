# Reporting — クライアント報告書パターン

Deliveryフェーズで読む。にからすのクライアント報告の型。

## インターフェース

**入力**: 進捗データ、成果物、KPI実績、次アクション案
**出力**: クライアント報告書（docx/pptx）、トンマナ適用済み
**次のフェーズ**: なし（デリバリー完了）or Planning（却下時）

---

## 1. 構成パターン

### パターンA: 定期報告（月次/週次）

```
1. サマリ（3行以内で「今月はこうでした」）
2. 進捗（完了→進行中→遅延・課題の順。ポジティブ面を先に）
3. 数値実績（KPIダッシュボード形式）
4. 次月のアクション
5. ご相談事項（意思決定が必要な項目）
```

### パターンB: 提案資料

```
1. エグゼクティブサマリ（1枚で全て伝える）
2. 背景・課題（クライアントが感じている痛みを言語化）
3. 提案する解決策（What + Why + How）
4. 期待効果（定量で示せるものは必ず数値で）
5. スケジュール・ロードマップ
6. 費用
7. 想定リスクと対策
8. Next Steps
```

### パターンC: 調査報告

```
1. エグゼクティブサマリ（最重要ファクト3つ）
2. 調査の背景・目的
3. 調査方法
4. 発見事項（本編）
5. 仮説検証結果（支持/修正/棄却の表）
6. 推奨アクション（優先順位付き）
7. 残存リスク・不確実性
```

## 2. トンマナルール

### 共通原則

- **結論を先に**: 背景から入るのはNG
- **数字で語る**: 「大幅に改善」→「74.7%から97.6%に改善（+22.9pt）」
- **課題は解決策とセットで**: 課題だけ並べると不安を煽る

### クライアント別トンマナ

PJ固有上書きファイル（`.claude/skills/bzrde-methodology/references/client-tonmana.md`）で管理。以下は既知パターン:

- **ExaWizards/関電**: 紺色左バー(#1A3C7A)、白背景、Calibri、Confidential表記
- **Aiful/Kiei経由**: Confidentialヘッダー、Yu Gothic、docx形式
- **awabar/Crosstech**: Notion+Slack、カジュアルな文体OK
- **杉江商会**: カフェで先輩に教えてもらう感じ。ガチプレゼン禁止。テック用語（MCP、RAG、API、LLM）禁止。データはグラフや表で視覚的に。金額はスライドに載せず口頭で。ティール系配色(#0D9488)

## 3. 感情設計

### 冒頭で「安心」を与える

最初に「順調です」「計画通りです」「成果が出ています」のいずれかを伝える。課題がある場合でも、まずポジティブな面を先に。

### 課題の伝え方

❌ 「〇〇が遅延しています」
✅ 「〇〇は当初計画から1週間遅延していますが、△△の対策により来週中に回復見込みです」

### 提案の伝え方

クライアントが「自分で選んだ」と感じるように設計する。3つの選択肢を提示し、推奨を明示しつつ、最終判断はクライアントに委ねる。

### 感情曲線設計（提案・セールスセッション向け）

クライアントに「やりたい」と自ら言わせるための感情設計手法。定期報告では不要。提案資料・セールスセッション・デモで使う。

#### 6ステップ

1. **ペルソナ再構築**: クライアントの人柄、意思決定スタイル、刺さるポイント、地雷を構造化
2. **至ってほしいアクションの定義**: 「クライアントが〇〇と発言し、△△の日程調整に入る」レベルで具体的に
3. **最終的な感情状態の定義**: アクションに至るために必要な感情の組み合わせ（例: 確信＋自信＋期待）を1文で
4. **感情曲線の設計**: セッションの時間軸に沿って感情の変曲点を配置。山と谷を意図的に作る
5. **トリガー設計**: 各変曲点で「何を見せる/言う/聞くと、その感情になるか」を特定
6. **トリガー×提案項目の紐付け**: トリガーと、提案書の個別項目（機能・価格・スケジュール等）を対応付ける

#### 感情曲線の典型パターン（2時間セッション）

安心 → 驚き → 自分事化 → 体感/自信 → 確信蓄積 → ワクワク → 決断

- **安心**: アイスブレイク。構えを解く
- **驚き**: 最初のデモで掴む。「え、こんなことできるの？」
- **自分事化**: 「うちでもこれできる？」。驚きが自社の課題と接続
- **体感/自信**: クライアント自身が触って「できた」と感じる
- **確信蓄積**: 実績・活用事例の共有。「この人たちは本物だ」
- **ワクワク**: ビジョン共有。「そんな未来が来るのか」。最大の山
- **決断**: 具体的ロードマップで不安をゼロに。「どう思いますか？」で待つ

#### 原則

- **こちらから押すな。クライアントが自ら言う設計にしろ**: 「お願いします」ではなく「どう思いますか？」で待つ
- **金額は口頭で伝えろ**: スライドに書くと交渉余地がなくなる。反応を見て調整できる余地を残す
- **3選択肢で「自分で選んだ」感覚を作れ**: 推奨を明示しつつ、最終判断はクライアントに

#### 具体例

**案件**: 杉江商会 来期契約提案セッション（2026-03-24）
**ペルソナ**: 31歳後継ぎ社長、AI敬遠だが興味あり、社員を教えたいモチベ、ITリテラシー低
**至ってほしいアクション**: 「来期もお願いします。いつから始められますか？」と自発的に発言
**最終感情状態**: 確信＋自信＋期待
**トリガー数**: 9（デモ3+体験1+にからす活用共有1+番頭ストーリー1+ロードマップ1+プラン提示1+クロージング1）
**結果**: 3パッケージ(A:140万/B:190万推奨/C:250万)を感情曲線に紐付けたスライド11枚+デモパッケージ

## よくある失敗

| パターン | 帰結 | 修正策 |
|---------|------|--------|
| 課題を対策なしで報告する | クライアントが不安になる | 課題には必ず対策をセットで |
| 背景から入る（結論先行でない） | 結論にたどり着く前に読者が離脱 | サマリを最初に。背景は2ページ目 |
| クライアントのトンマナを無視する | プロ意識を疑われる | PJ固有上書きファイルを必ず確認 |
| 数値を曖昧語で表現する | 信頼性が低下 | 「改善した」→具体数値+差分 |

## このフェーズでやらないこと

- 新規仕様の策定（Planningに差し戻す）
- 追加リサーチ（Researchに差し戻す）
- クライアントへの約束なしの納期変更

## 品質基準

- クライアントのトンマナに完全準拠
- 数値は全てソースを明記
- 課題には必ず対策がセット
- 次のアクションが「誰が、何を、いつまでに」で定義
- 1枚で全体像が把握できるサマリがある
