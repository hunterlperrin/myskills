# GitHub公開リポジトリ検索スキル

ユーザーの要求に近いことを実現できるGitHub公開リポジトリを検索する。
READMEを読んで「ローカルですぐ試せるか」を判定し、URLと超短文サマリを返す。

## 手順

### Step 1: キーワード生成（10個）

`$ARGUMENTS` から検索キーワードを **10個** 生成する。以下の5つの軸 × 2パターンで網羅する:

| 軸 | 説明 | 例（「画像をWebPに一括変換したい」の場合） |
|----|------|----------------------------------------|
| **そのもの語** | やりたいことの直接的な英語表現 | `webp converter`, `batch image convert webp` |
| **上位概念語** | より広いカテゴリ・ドメイン | `image optimization`, `image processing tool` |
| **実装アプローチ語** | 実現手段・技術的アプローチ | `image compression cli`, `sharp webp` |
| **類似ツール語** | 既存の有名ツール名や代替表現 | `imagemagick alternative`, `squoosh cli` |
| **課題・ユースケース語** | 解決したい課題やシーン | `optimize images for web`, `reduce image size` |

### Step 2: 並列検索

生成した10個のキーワードで `gh search repos` を**並列実行**する。Agentツールを使って並列化すること。

各キーワードに対して以下のコマンドを実行する:

```bash
gh search repos "<キーワード>" --sort=stars --order=desc --limit=5 --json url,description,stargazersCount
```

#### 検索の工夫

- **スター数フィルタ**: ノイズが多い場合は `--stars=">50"` を追加して品質を担保する
- **言語フィルタ**: ユーザーの要求に特定言語が含まれる場合は `--language=<lang>` を追加する
- **更新日フィルタ**: メンテされているものを優先したい場合は `--updated=">2024-01-01"` を追加する
- **完全一致**: フレーズ検索が必要な場合はキーワードをダブルクォートで囲む

### Step 3: 結果の統合と重複排除

1. 全検索結果を統合する
2. URL単位で重複を排除する
3. 複数のキーワードでヒットしたリポジトリは関連度が高いと判断し優先する
4. スター数・説明文の一致度で上位 **10〜15件** に絞る（次のステップで精査するため多めに残す）

### Step 4: README精査 — 「すぐ試せるか」判定

上位候補のREADMEを**並列で**取得して精査する。Agentツールで並列化すること。

#### 取得方法

以下の優先順で取得する:
```bash
# 方法1: gh CLIで取得（最も確実）
gh api repos/<owner>/<repo>/readme --jq '.content' | base64 -d

# 方法2: WebFetchで取得
# https://raw.githubusercontent.com/<owner>/<repo>/HEAD/README.md
```

READMEが存在しない場合は、リポジトリのトップレベルファイル一覧を確認する:
```bash
gh api repos/<owner>/<repo>/contents --jq '.[].name'
```
`GETTING_STARTED.md`, `docs/`, `INSTALL.md` などがあればそれを読む。

#### 判定基準

READMEから以下を読み取り、各リポジトリに**すぐ試せる度**を付ける:

| 判定項目 | ○（すぐ試せる） | ×（すぐ試せない） |
|---------|----------------|-----------------|
| **インストール** | `npm install` / `pip install` / `brew install` / バイナリDL / `git clone` して即実行 | ビルドに複雑な依存、Docker必須、OS限定 |
| **API依存** | APIキー不要、またはオプション | 必須のAPIキー（OpenAI, AWS等）がないと動かない |
| **データ依存** | サンプルデータ付き、または任意のローカルファイルで動く | 特定のDB・データセット・外部サービスが前提 |
| **設定量** | 設定なし or 最小限の設定ファイル | 大量の環境変数や設定ファイルが必要 |
| **動作確認** | READMEにクイックスタート・使用例がある | 使い方の記載が薄い・不明確 |

**すぐ試せる度の定義:**
- ⚡ **即試行可**: clone/installして1分以内に動かせる。API・データ依存なし
- ✅ **容易**: 数ステップで動く。軽い設定やサンプルデータ準備のみ
- ⚠️ **条件付き**: APIキーや特定環境が必要。動かすまでに準備が要る
- ❌ **困難**: 複雑な依存・必須の外部サービス・大規模データが前提

### Step 5: 出力

以下のフォーマット**のみ**を出力する。余計な説明は不要。

**⚠️・❌のリポジトリは原則として結果から除外する。** 候補が少なすぎる場合（⚡・✅が3件未満）のみ、⚠️を補足付きで含める。

```markdown
## 検索結果: [要求の要約]

| # | リポジトリ | ★ | 試 | 概要 |
|---|-----------|---|---|------|
| 1 | [owner/repo](URL) | 1.2k | ⚡ | 一言サマリ |
| 2 | [owner/repo](URL) | 800 | ✅ | 一言サマリ |
| 3 | [owner/repo](URL) | 200 | ✅ | 一言サマリ |
| ... | | | | |

> 試 = すぐ試せる度（⚡即試行可 / ✅容易 / ⚠️条件付き）
```

## ルール

- 出力はURLと超短文サマリのみ。長い説明は書かない
- 一言サマリは日本語で、15文字以内を目指す
- ⚡・✅のリポジトリを優先し、⚠️・❌は原則除外する
- アーカイブ済み・明らかにメンテされていないリポジトリは除外する
- 検索結果が少ない場合はキーワードを緩めて再検索する
- 検索結果が多すぎる場合はスター数フィルタを厳しくする
- 日本語で応答する
